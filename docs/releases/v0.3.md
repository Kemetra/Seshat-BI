# Seshat BI -- v0.3 release candidate

This release candidate records the repository state prepared from `main` at
`d51f0b8` on 2026-07-14. It is a repository candidate only: it does not prove
PyPI, GitHub Release, Claude catalog, Codex catalog, or OpenAI public-plugin
availability. Each of those actions requires its own named-owner approval and
external evidence.

## Product identity

Seshat BI remains an **agent-first BI readiness system**. It requires evidence
and named-human approvals at readiness gates; it does not turn an agent
recommendation, a version number, or a release-candidate audit into an
approval.

## What this candidate adds since v0.2.0

- **Governed existing-PBIP-project adoption**: a module that adopts an
  already-authored PBIP project into the governance model. Evidence: `a7cba72`
  (PR #271).
- **Two P2 fixes on the PBIP adoption module**: literal Power Query M
  data-source detection closes a gap in the existing C1 boundary rule, and a
  new source-reference inventory records each table's partition/M source for
  review. Evidence: `f132d28` (PR #279).
- **Specs 127-131**, ratified 2026-07-14 (`388f8ac`, PR #280) and implemented
  together (`d51f0b8`, PR #281):
  - **Spec 127 -- Shareable Seshat Proof (showcase bundle)**: composes existing
    Explorer, Passport, readiness, review, blocker, approval, and lineage
    evidence into a disclosure-safe static offline bundle. Skill/composer-only;
    no new CLI verb.
  - **Spec 128 -- Public Extension-Pack Catalog** (`seshat pack search /
    inspect / add`): community discovery over the existing declarative pack
    system, backed by a reviewed static git registry with hash/schema
    verification and fail-closed handling of invalid or tampered packs.
  - **Spec 129 -- Agent Compatibility Certification** (`seshat agent verify`):
    a new CLI verb certifying agent/tool compatibility; output stays
    local-only.
  - **Spec 130 -- Friendly PR Reviewer**: a plain-language summary layer over
    existing PR review evidence.
  - **Spec 131 -- Portfolio Watch** (`seshat watch build`): a recurring,
    read-only portfolio summary aggregating drift, stale approvals, changed
    readiness, and blockers into one prioritized next action per governed
    scope. Agent-/skill-driven, mirroring `retail-control-room`.
- **Coordinated release preparation workflow**: an owner-triggered
  `workflow_dispatch` action (`.github/workflows/prepare-coordinated-release.yml`)
  that synchronizes the version across Python packaging, the Claude
  marketplace/plugin manifests, and the Codex plugin manifest into one draft
  release PR. Evidence: `ea70659` (PR #278).
- **`speckit-batch` JSON-string `args` tolerance**: the batch runner now
  accepts `args` passed as a JSON-encoded string, not only a native
  array/object. Evidence: `11c500e` (PR #277).

## Documentation and release discipline

This candidate is the first to be prepared via the coordinated release
workflow added in PR #278: one draft PR synchronizes `pyproject.toml`,
`CHANGELOG.md`, the Claude marketplace metadata, and both generated Claude/
Codex bundles, rather than each surface being hand-edited independently.

## What remains deliberately outside this candidate

- No tag, GitHub Release, package-index publication, catalog publication, or
  OpenAI public-plugin submission has been performed.
- Public installation evidence for Python, the Claude repository flow, and the
  Codex repository flow remains unverified until owner-authorized external
  acceptance is recorded.
- The Power BI execution adapter (F016) remains deliberately deferred and
  execution-only; it is not unlocked by this candidate.

## Release criteria

The owner must review the exact candidate revision and generated artifact
digests, then make separate action-scoped decisions for tag creation, package
publication, GitHub Release publication, and any catalog or public-plugin
submission. See
[`docs/operations/release-acceptance-checklist.md`](../operations/release-acceptance-checklist.md)
and [`docs/operations/versioning-policy.md`](../operations/versioning-policy.md).

Note on classification: PR #279's C1 boundary-rule fix broadens what the
*existing* C1 rule detects (it now also catches literal Power Query M
data-source calls it previously missed), which can newly flag an
already-committed PBIP project that was previously passing. The owner has
classified the overall bump MINOR (0.3.0) on the strength of specs 127-131
being additive new capabilities/CLI verbs.
